## 十、关于"从 Python 到纯 Go 彻底改造"的结论

> 更新于: 2026-04-17 02:00

### 决策：不另起炉灶，在 selfmind 现有代码基础上演进

**放弃的方案**：在 `/mnt/d/wwwroot/workspace/hermes-go` 另起纯 Go 项目。

**理由**：
- selfmind 当前已有完整的 Go 基础设施（cobra、bubbletea、modernc.org/sqlite、gin/echo 选型已验证）
- 20+ 工具、Agent 核心、TUI、内存管理、FTS5 搜索均已实现
- 从头重建意味着丢弃所有已验证的代码，半年内没有可运行产出

**保留的参考**：gemini-3-flash 提出的"五步演进法"作为架构指导：

| 五步法内容 | 对 selfmind 的价值 | 当前状态 |
|-----------|-------------------|---------|
| 统一事件协议（Protobuf/JSON） | 长期需要，跨语言 RPC 时加 | JSON 当前够用 |
| 流式 LLM 输出（StreamProcessor） | **值得补，selfmind 无流式处理** | 缺失 |
| Per-session SQLite 隔离 | 参考方向，当前多租户共享 DB 够用 | 已有基础 |
| Debouncer 消息聚合 | **值得加，防止工具调用刷屏** | 缺失 |
| 多端 Provider 接口 | 与 selfmind 架构一致 | WeChat stub 有待验证 |
| ACP IDE 集成 | 长期目标 | 无 |

**五步法中值得借鉴到 selfmind 的具体点**：
1. **StreamProcessor** — 阶段 3 优先实现
2. **Debouncer** — 阶段 3 优先实现
3. **PTY 持久化** — 阶段 4 核心能力

### 最终演进路径

```
selfmind 现有代码
  ↓ 阶段1: PatchTool/ClarifyTool 接入（明天）
  ↓ 阶段2: memory 工具 + 危险命令检测
  ↓ 阶段3: StreamProcessor + Debouncer
  ↓ 阶段4: PTY 持久化（跨端状态恢复）
  ↓ 阶段5: 工具生态补全
  ↓ 阶段6: CLI/UX 提升
  ↓ 阶段7: ACP / IDE 集成
```

每一步都有**可运行的产出**，而非憋一个半年的大招。
